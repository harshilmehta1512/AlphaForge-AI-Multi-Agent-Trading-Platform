# Dashboard Specification

This document describes how the first-phase dashboard should look and how its pages should connect together.

The dashboard is part of the product, not just a developer convenience. It should make the end-to-end trading flow understandable to a user without requiring them to inspect backend logs or code.

## Dashboard Goals

The dashboard should allow a user to:
- choose a tracked stock
- view the latest recommendation
- inspect agent-by-agent reasoning
- see whether risk changed the result
- see the simulated trade and portfolio effect
- review historical recommendation and execution summaries

The dashboard should make the system understandable for three kinds of viewers:
- a teammate building the product
- a reviewer or judge evaluating the product
- a future user who wants to trust the recommendation flow

## Recommended MVP Pages

## 1. Login Page

### Purpose
Entry point into the application.

### Phase-One Expectation
If authentication is implemented, keep it simple. If auth is not ready, a temporary development access flow is acceptable for the MVP.

### Connection
After login, the user lands on the main dashboard overview.

## 2. Dashboard Overview Page

### Purpose
Provide a high-level snapshot of the system.

### What Should Be Visible
- tracked ticker selector
- latest final recommendation for the selected ticker
- final confidence
- agent contribution summary
- recent simulated trade summary
- recent alert or risk status

### Additional Recommended Elements
- timestamp of latest evaluation
- one-click navigation to deeper detail pages
- quick explanation snippet such as why the current signal exists

### Why This Page Exists
This is the main entry page for both demos and daily use.

## 3. Recommendation Detail Page

### Purpose
Show the full reasoning behind one recommendation.

### What Should Be Visible
- ticker
- final signal
- final confidence
- final rationale
- each agent's signal
- each agent's confidence
- each agent's contribution percentage
- risk override, if any

### Why This Page Is Critical
This is the most important explainability page in the product. If the team has limited UI capacity, this page matters more than cosmetic enhancements elsewhere.

### Connection
The dashboard overview should link into this page when the user clicks a recommendation card or ticker row.

## 4. Agent Breakdown Page

### Purpose
Provide a deeper look into individual agent outputs for a selected ticker.

### What Should Be Visible
- market data agent rationale
- news agent rationale
- sentiment agent rationale
- fundamental agent rationale
- timestamps and data windows used by each agent

### Recommended Presentation
Each agent should be shown in a separate clearly labeled panel with:
- signal
- confidence
- rationale
- key factors
- data window used

### Why This Page Exists
This is the clearest place to demonstrate explainability and debug disagreements between agents.

## 5. Portfolio Page

### Purpose
Show the paper-trading portfolio state.

### What Should Be Visible
- current simulated positions
- cash or available simulated capital
- latest allocation decisions
- simple portfolio summary

### Recommended MVP Details
- per-ticker paper position
- last update timestamp
- simple cumulative paper exposure summary

### Phase-One Boundary
Keep this simple. It is not a full portfolio analytics terminal.

## 6. Trade History Page

### Purpose
Show simulated trades that were executed by the system.

### What Should Be Visible
- ticker
- action
- quantity or paper size
- simulated execution price if tracked
- execution timestamp
- originating recommendation reference

### Why This Page Matters
Without a trade history page, the product story stops at recommendation. This page proves that a downstream action happened, even if only in simulation.

## 7. Reports Page

### Purpose
Show readable summaries generated by the reporting layer.

### What Should Be Visible
- recommendation summaries
- trade summaries
- risk-triggered overrides
- recent system activity

### Phase-One Constraint
Keep this readable and simple. It should summarize, not overwhelm.

## Suggested Page Connections

```text
Login
-> Dashboard Overview
-> Recommendation Detail
-> Agent Breakdown
-> Portfolio
-> Trade History
-> Reports
```

The overview page should act as the hub. Recommendation detail should connect naturally to agent breakdown. Portfolio and trade history should be reachable from the main navigation.

## Navigation Model

The MVP navigation should be simple and stable:
- left sidebar or top nav
- overview as default landing page
- ticker selection available globally or clearly on the overview
- detail pages linked from overview cards or tables

## Dashboard Data Dependency Map

### Overview Page Depends On
- final recommendation endpoint
- summary reporting output

### Recommendation Detail Depends On
- final recommendation endpoint with full explainability payload

### Agent Breakdown Depends On
- either the same final recommendation payload or per-agent inspection endpoints

### Portfolio Page Depends On
- portfolio state output from backend

### Trade History Page Depends On
- simulated trade history output from backend

### Reports Page Depends On
- reporting summary output from backend

## MVP UI Priority

If time becomes tight, build pages in this order:
1. Dashboard Overview
2. Recommendation Detail
3. Agent Breakdown
4. Portfolio
5. Trade History
6. Reports

If auth is not essential for the final demo, treat login as low priority.

## Dashboard Non-Goals For Phase One

Do not let the dashboard scope expand into:
- advanced charting terminals
- portfolio analytics comparable to institutional platforms
- highly dynamic streaming views that require WebSockets
- generic admin settings pages with no impact on the MVP story
